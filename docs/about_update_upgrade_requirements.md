# Tube_Ultimate_Player “关于 / 版本检测 / 在线升级 / JS Runtime 安装”方案备案

## 1. 背景

当前项目已经具备：

- 顶部导航与多页面切换框架
- 设置页面与配置持久化
- 下载管理与进度展示能力
- GitHub Release 发布链路
- 应用图标素材与 `.ico` 文件

本次需求希望在此基础上补齐：

- “关于”页面
- 远程版本检测
- Release Note 展示
- 在线升级
- JS Runtime 缺失提示与自动安装
- 统一图标应用

## 2. 需求理解与完善

### 2.1 关于页面

在顶部导航中新增“关于”按钮，放在“设置”之后。

“关于”页面展示以下信息：

- 应用图标
- 应用名称
- 当前版本
- 应用简介
- GitHub 仓库链接
- “检测版本”按钮
- “在线升级”按钮
- 当前检测状态 / 最新版本状态
- 最新版本 Release Note 展示区

补充建议：

- “在线升级”按钮默认禁用，只有检测到新版本后才激活
- 展示“当前版本 / 最新版本 / 发布日期”
- GitHub 链接使用可点击按钮，而不是纯文本
- Release Note 使用只读文本区，便于滚动查看

### 2.2 版本检测

版本检测建议以 GitHub Releases 为唯一远端版本源。

检测逻辑：

1. 读取本地 `app_version.txt`
2. 请求 GitHub Releases 最新版本
3. 解析最新版本号、发布时间、Release Note、下载资源
4. 与本地版本比较

补充建议：

- 默认优先读取“最新非 draft release”
- 可接受 `prerelease`
- 版本比较按语义字符串比较规则处理，兼容 `0.1.0-dev`
- 若远端获取失败，提示网络或 GitHub API 异常

### 2.3 在线升级

在线升级分两类：

1. 便携版
2. 安装包版

升级前要先判断本地运行形态。

建议判断规则：

- 若当前进程位于解压目录、旁边存在 `3rdpart/`、`README.md`、`app_version.txt` 等便携结构，则视为便携版
- 若当前进程位于安装目录，或存在典型安装包路径特征，则视为安装版
- 若无法可靠判断，则给出“请选择下载便携版或安装包”的兜底提示

#### 便携版升级

流程建议：

1. 下载 GitHub Release 中的便携版 zip 到运行时目录的临时升级目录
2. 显示下载进度、速度、已下载大小
3. 下载完成后提示：
   “已下载完成，请关闭当前应用后执行替换升级”
4. 第一版不直接热替换当前运行目录

原因：

- 当前进程仍占用 exe 和 dll
- Windows 下边运行边替换容易失败
- 第一版以“下载完成 + 引导关闭替换”为主，更稳

补充建议：

- 下载完成后提供“打开下载目录”或“打开升级包目录”
- 可为后续二期预留“退出后自动替换”的外部 updater 方案

#### 安装包升级

流程建议：

1. 下载 GitHub Release 中的安装包 exe 到临时升级目录
2. 显示下载进度、速度、已下载大小
3. 下载完成后提示：
   “安装包已下载完成，请关闭当前应用后执行安装”
4. 第一版不在应用运行中直接启动安装器

原因：

- 安装包通常会覆盖当前运行文件
- 运行中启动安装器体验不稳定
- 先保证下载与引导流程可靠

### 2.4 JS Runtime 缺失提示与自动安装

设置页面当前已有 JS Runtime 选择项，但没有明确的缺失提示和自动安装入口。

建议增强为：

- 显示当前检测结果：`已检测到 Node.js` / `未检测到 JS Runtime`
- 若未检测到，则显示提示文案和“安装 Node.js”按钮

自动安装方案建议：

#### 第一优先方案

跳转官方 Node.js LTS 下载页，并提示用户完成安装后重新启动应用。

优点：

- 实现稳定
- 无需管理员权限处理逻辑
- 合规与可维护性最好

#### 第二优先方案

在 Windows 上静默下载 Node.js 官方 MSI 或 EXE 安装包到临时目录，并调用系统安装器启动。

风险：

- 需要管理员权限
- 企业环境可能被拦截
- 不同版本的安装参数维护成本高

本次建议：

- 第一版采用“自动下载安装器并启动系统安装程序”的半自动方案
- 如果启动安装失败，则回退为打开官网页面

这样既满足“点击安装”的用户预期，也保留较稳的失败回退路径。

### 2.5 图标应用范围

用户需求里提到“将设计好的图标应用到各个标题栏、安装包和桌面图标”。

建议覆盖：

- 主窗口标题栏图标
- 应用内“关于”页面图标
- PyInstaller 生成的 exe 图标
- 安装包图标
- 安装后的桌面快捷方式图标

现状补充：

- 仓库中已有：
  - `docs/assets/icons/app-icon.ico`
  - 多尺寸 png/svg 图标
- `build_installer.py` 当前已将 `.ico` 传给 PyInstaller
- 但安装包 `installer.iss` 仍建议明确指定安装器与卸载图标

## 3. 实现方案

### 3.1 新增页面与导航

新增：

- `ui/about_page.py`

在 `ui/main_window.py` 中：

- 增加 `about_nav`
- 将 `AboutPage` 加入 `QStackedWidget`
- 放置到“设置”后面

### 3.2 新增服务层

建议新增：

- `services/update_service.py`
- `workers/update_check_worker.py`
- `workers/update_download_worker.py`
- `services/runtime_install_service.py`

职责划分：

- `UpdateService`
  - 读取本地版本
  - 拉取 GitHub Release 元数据
  - 比较版本
  - 挑选便携版 / 安装包资产

- `UpdateCheckWorker`
  - 异步执行版本检查

- `UpdateDownloadWorker`
  - 异步下载升级包
  - 回传下载进度

- `RuntimeInstallService`
  - 检测 JS Runtime
  - 下载 Node.js 安装包
  - 启动安装器
  - 失败时回退打开官网

### 3.3 版本数据来源

远端版本信息建议读取：

- GitHub API `releases/latest`

需要获取字段：

- `tag_name`
- `name`
- `published_at`
- `body`
- `assets`

便携版资产匹配规则：

- 文件名包含 `portable`

安装包资产匹配规则：

- 文件名包含 `setup` 或 `installer`

### 3.4 升级包下载落地路径

所有下载内容放到：

- `%LocalAppData%\\Tube_Ultimate_Player\\updates`

建议结构：

- `updates/latest_portable.zip`
- `updates/latest_installer.exe`
- `updates/nodejs-installer.exe`

### 3.5 进度展示

关于页面中增加：

- 状态标签
- 进度条
- 已下载大小 / 总大小

若现有 Toast 不适合长任务展示，则在关于页内嵌下载状态区，不额外弹窗刷屏。

### 3.6 设置页 JS Runtime 提示

在 `ui/settings_page.py` 中新增：

- 运行时检测结果标签
- “安装 Node.js”按钮
- “打开官网”按钮

交互建议：

- 若已检测到 `node` / `deno` / `bun` / `quickjs`，隐藏安装按钮
- 若未检测到，显示安装提示

### 3.7 图标落地

建议新增统一资源入口，避免散落硬编码：

- `resources/icons/app-icon.ico`
- `resources/icons/app-icon.png`

也可以继续沿用 `docs/assets/icons/`，但更推荐将运行时真实依赖迁移到 `resources/icons/`。

本次编码建议顺手完成：

- 主窗口 `setWindowIcon`
- 关于页图标显示
- `installer.iss` 中的安装器图标与快捷方式图标指定
- 若需要，README / 资源路径同步调整

## 4. 风险与边界

### 4.1 第一版不做热替换

不建议第一版直接在应用运行中替换 exe / dll。

原因：

- Windows 文件锁
- 替换失败后易留下半升级状态
- 需要单独 updater 进程才稳

### 4.2 Node.js 自动安装存在权限依赖

如果用户系统限制安装器执行，自动安装可能失败。

因此必须提供：

- 自动下载并启动安装器
- 失败时打开官网页

### 4.3 GitHub API 访问失败

需要明确提示：

- 网络不可用
- API 被限流
- Release 不存在

### 4.4 版本判断

`0.1.0-dev` 这类版本带预发布后缀，建议做统一比较函数，不要直接字符串大小比较。

## 5. 编码拆分建议

建议分 4 步实施：

1. 关于页 + 导航接入
2. 版本检测 + Release Note 展示
3. 在线升级下载与进度展示
4. JS Runtime 缺失提示 + Node.js 安装入口 + 图标完善

## 6. 验证方法

### 6.1 关于页

- 点击“关于”按钮可以进入页面
- 页面显示应用图标、名称、版本、简介、GitHub 链接

### 6.2 版本检测

- 本地版本低于远端版本时，提示有新版本
- 能展示远端 Release Note
- 本地版本等于远端版本时，提示已是最新版本
- 网络断开时，给出错误提示

### 6.3 在线升级

- 便携版模式下可下载 zip
- 安装版模式下可下载安装包 exe
- 下载过程中进度条持续更新
- 下载完成后出现“关闭当前应用后升级/安装”的提示
- 文件落到 `%LocalAppData%\\Tube_Ultimate_Player\\updates`

### 6.4 JS Runtime 安装

- 未检测到运行时时，设置页出现提示与安装按钮
- 点击安装后：
  - 成功下载并启动安装器，或
  - 失败时打开 Node.js 官网

### 6.5 图标

- 主窗口标题栏图标正确
- 生成的 exe 图标正确
- 安装包图标正确
- 安装后桌面快捷方式图标正确

## 7. 本次建议结论

建议按以下原则进入编码：

1. 关于页作为独立页面接入主导航
2. 远端版本源统一使用 GitHub Releases
3. 第一版在线升级只做到“下载完成并引导关闭应用后升级”
4. JS Runtime 安装采用“下载官方安装器并启动，失败则打开官网”的半自动方案
5. 图标统一应用到主窗口、打包 exe、安装器与快捷方式
